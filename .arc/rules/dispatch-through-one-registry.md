When a program selects among a closed set of interchangeable implementations by name or tag — backends, providers, drivers, formats, strategies — give the set one home: a registry mapping each name to its constructor or behavior, with every per-implementation difference (its defaults, the capabilities it honors, the ones it must refuse) behind one shared interface. Derive every operation over the set — dispatch, the known-names listing, membership validation, error wording — from that one table, so the rest of the code drives "the selected one" without branching on which it is. The anti-pattern is a name list kept beside a parallel switch on that name, edited in lockstep by hand and scattered across call sites: adding a member means finding every branch, a missed site silently mishandles the new member, and per-member policy leaks into callers as conditionals. A registry makes adding a member one row, and makes it impossible for the enumeration and the behavior it selects to disagree. This is distinct from exhaustively matching a closed enum at a single decision point (which is fine); it targets the same name or tag branched on at more than one site. Flag a name or type tag switched on in multiple places to select behavior, or a set's membership maintained separately from the per-member behavior it drives.
