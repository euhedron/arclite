A destructive, multi-step replacement of a hard-to-recreate resource — overwriting a running binary, swapping a live file in place, any "move the old aside, move the new in" sequence — must be ordered so a failure at any step leaves a usable prior state, and must actively roll back a partial change rather than return mid-mutation. Do not remove or displace the original until the replacement is staged and ready; if the final swap fails, restore the original before propagating the error; and delete partial artifacts (a half-written download, a staging file) on the failure path. When even the rollback fails — the resource is now in a state only the user can repair — surface it loudly, naming the exact recovery path (where the salvaged original is and how to restore it), rather than swallowing it. Flag a multi-step replacement of a critical resource whose error path can strand it missing or half-written, or that abandons staging artifacts on failure.
