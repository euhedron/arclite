An externally-supplied identifier that becomes part of a filesystem path — a CLI argument, request parameter, stored id, filename, or key originating outside the program — must be validated to a single safe path segment at the boundary where it first enters, before it is joined to any base directory. Reject the empty value, path separators, parent-directory (`..`) components, and absolute or drive/scheme-qualified forms, so a crafted identifier cannot escape the directory it is meant to address. Joining a trusted base to an unvalidated identifier is a directory-traversal vector even when the value "should" always be well-formed; validating once at entry lets the downstream joins, reads, and writes treat it as safe. Flag an external identifier interpolated into a path with no such guard at its entry boundary — not a downstream join of a value already validated upstream or generated internally, where a redundant re-check would be needless defensiveness against an input that cannot occur.
