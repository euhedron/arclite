Treat the current state of the code as an input to question, not a constraint to optimize within. That something already exists — a value already in the file, an existing file, the prevailing convention, the way it is currently set up — describes what is, not what should be, and is no justification for a design choice. Decide each question on its merits: what is the right home, shape, and structure for this data or behavior? Then change whatever must change to match — a new line or file, a renamed convention, the whole structure rather than only the leaf being added. Flag choices defended by precedent or convenience over fit, and additions contorted to preserve surrounding structure that should itself have moved.
