When recording or reporting what an operation actually did — the model that answered, the version invoked, the endpoint reached, the amount spent — derive every reported fact from the operation's own result (the response payload, measured usage, observed outcome), never by echoing back the parameters that were requested. The two can silently diverge: a service substitutes or falls back, an alias resolves to a concrete build, a value is capped or coerced downstream — a request is an intention, not an outcome, and echoing it mislabels the record for everyone who later trusts it (attribution, cost accounting, provenance, audit). Resolve the identity from what the result confirms actually ran; fall back to the requested value only when the result genuinely names none, and treat that fallback as a disclosed limitation, not the norm. Flag a report, log entry, receipt, or provenance record that presents a requested parameter as the realized outcome where the result carries the true, possibly different value.
