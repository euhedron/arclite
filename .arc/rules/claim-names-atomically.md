When code creates a uniquely-named resource — a file, directory, lock, or keyed record — in a location where other processes or sessions may create one concurrently, claim the name with a single atomic create-if-absent operation that fails when the name already exists, and react to that failure (advance to the next candidate, retry, or report the conflict). Splitting the claim into an existence check followed by a separate create opens a time-of-check-to-time-of-use window: two creators both observe the name free, and one silently overwrites the other's work. The guarantee must come from the create operation's own exclusivity, not from an assumption that names are unique or that a collision is unlikely. Where only a single creator can ever run the code, a check-then-create is acceptable; this targets shared locations with concurrent creators. Flag an acquisition of a unique name or resource that tests for absence and then creates in a separate step where concurrent creation is possible.
