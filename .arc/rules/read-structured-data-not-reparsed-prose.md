When your own program produces or stores structured data — a language model's result, facts you record (notes, statuses, relationships, provenance), an external payload you persist — represent and read it as structure, never round-trip it through prose that a later consumer reparses. Take a model's structured result through a forced schema (a tool or emit the model must call, validated so it retries on a mismatch) and read the typed fields; never recover them by regex or string-splitting the model's free text, and never keep a prose-parsing fallback beside the structured path. Store queryable facts as first-class rows or columns — or keep the raw payload verbatim as the source of truth and derive projections from it — rather than splicing them into a free-text blob that one writer formats and a scattered set of parsers reconstruct. Prose is a presentation layer; rebuilding fields from it is an implicit contract between a formatter and every reparser that silently drops or mangles exactly the data driving the logic, leaving the record that steers the system parsed rather than read. A defined serialization format accessed through its one owning reader/writer — a delimited metadata header, a config format, a wire format — is structure, not prose, even though reading it necessarily parses text: the defect is recovering fields from free text that was formatted for presentation, or scattering ad-hoc parsers for one format across independent writers and readers. This concerns the structure of data the system itself emits and stores — separate from forcing a distinct tool the program calls into its machine-readable output mode. Flag a model result recovered by splitting or regexing its text, a structured path shadowed by a prose-parsing fallback, or a store that retains only a prose blob other code must reparse to recover structured facts.
